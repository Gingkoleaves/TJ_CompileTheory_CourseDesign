# 项目完成度汇报

## 评估依据

本报告根据课程作业 PPT 中的词法、语法规则要求，以及当前仓库源码和实测结果整理。

## 总体结论

当前项目已经完成类 Rust 语言的词法分析器和语法分析器主体，强制要求中的词法规则和最低语法规则基本覆盖。项目还能通过命令行和 Web 页面展示 token 与 AST，具备课程演示所需的基本形态。

不足主要集中在三类：

- 缺少完整语义分析，例如类型检查、变量声明检查、不可变变量赋值检查
- Web 服务当前只有编译验证，尚未补充接口自动化测试
- 部分扩展语法只有语法树构建，尚未配套语义检查

综合判断：若评分重点是“词法分析、语法分析和结果输出”，项目完成度较高；若要求进一步覆盖语义约束、实验报告完整性和鲁棒工程质量，则仍需补强。

## 源码结构分析

### `Easy_Lexer`

词法器结构清晰，核心入口是 `lex(input: &str)`。扫描过程采用手写逐字符读取方式，按标识符、数字、复合 token、单字符 token 的顺序处理。

优点：

- token 类型覆盖完整
- 支持行列号，便于错误定位
- 支持最长匹配，例如 `>=` 优先于 `>`
- 支持注释跳过和未闭合块注释报错
- 有针对关键字边界、复合符号和注释的单元测试

不足：

- `Cargo.toml` 中引入了 `axum`、`tokio`、`serde`、`serde_json`，但词法器 crate 本身并不需要 Web 依赖，依赖略重

### `Easy_Parser`

语法器是手写递归下降分析器。它先调用词法器得到 token，再解析为 `Program`、`FunctionDecl`、`Statement`、`Expr`、`TypeNode` 等 AST 节点。

优点：

- 覆盖强制语法规则较完整
- 表达式优先级处理清楚
- AST 结构便于后续展示和扩展
- 能报告语法错误及位置
- 单元测试包含最低语法和扩展语法样例

不足：

- `if` 语句在 AST 中以表达式语句形式出现，能工作，但和作业产生式中的“语句”概念不完全一致
- 缺少符号表和类型系统，不能进行变量、类型、可变性、返回值等语义检查

### `Easy_Server` 和前端

服务端使用 Axum 提供两个接口：

- `GET /`：返回根目录 `index.html`
- `POST /api/analyze`：接收源码，返回 token、AST 和错误信息

优点：

- 可以直观展示 token 表和 AST 树
- API 与分析器逻辑解耦较清楚
- 错误信息能返回到前端

不足：

- 前端依赖 CDN CodeMirror，离线演示可能受影响
- Web 服务没有自动化接口测试

## 主要风险和建议

已修复：

- AST 序列化问题已修复，数组和元组表达式现在使用结构体式枚举变体，例如 `Array { elements: Vec<Expr> }`
- 文件输出集成测试已统一 normalize 换行后再比较，避免 LF/CRLF 平台差异导致假失败

优先建议：

- 为 `Easy_Server` 增加至少一个 `/api/analyze` 接口测试

建议补强：

- 增加符号表，实现变量声明检查
- 增加基础类型检查，覆盖 `i32`、引用、数组、元组
- 检查不可变变量赋值错误
- 检查函数返回类型与 `return` 表达式是否一致
- 优化 crate 依赖，将 Web 相关依赖从 `Easy_Lexer`、`Easy_Parser` 中移除
- 在根目录建立 Cargo workspace，方便一次性构建和测试
