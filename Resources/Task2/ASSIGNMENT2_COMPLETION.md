## 未实现或与完整规则存在差距的点

1. **Range 不是一等普通表达式。**
   当前 `a..b` 主要作为 `for i in a..b {}` 的 iterable 解析；`let r = 0..3; for i in r { ... }` 会在 parser 阶段失败。该限制已由 `assignment2_ppt_matrix::optional_range_values_are_not_general_expressions` 记录。

2. **数组越界主要是静态诊断，不是运行时检查。**
   对 `a[3]`、`a[-1]` 等可静态求值的索引会报错；对 `a[i]` 这类动态索引只检查 `i` 的类型，不生成运行时边界检查四元式。

3. **数值类型只有整数。**
   当前 lexer/parser 没有浮点字面量类型；PPT 中类似 `n+0.1` 的“非整数”例子不会作为真正的浮点表达式进入类型系统，而会由现有词法/语法/字段访问路径报错。

4. **借用检查是课程级简化实现。**
   当前能检查同一词法作用域内根变量的可变/不可变借用冲突，但没有完整 Rust 生命周期、移动语义、重借用、字段级别别名分析和复杂逃逸分析。

5. **函数不是一等值。**
   函数名可被识别为函数项并给出友好错误，但不支持把函数作为变量保存、数组/元组元素保存或通过非标识符表达式调用。

6. **完整控制流返回分析有限。**
   当前能处理显式 `return`、尾表达式返回和若干常见缺失返回值场景，但没有做完整 CFG 级“所有路径必返回”证明。

7. **未覆盖更完整的 Rust 特性。**
   不支持 `&&`、`||`、字符串、字符、结构体、模式匹配、泛型、所有权移动、函数指针、闭包等；这些也不属于作业二 PPT 的最低要求。
